# Enterprise RAG AI Support Agent — Телеком AI-Ассистент

## Описание проекта

Система для автоматизации технической поддержки сотрудников оператора связи. Бот отвечает строго на основе загруженных внутренних документов (PDF-чанков), не выдумывает факты и всегда ссылается на источник из базы знаний. Поддерживает многоходовой диалог с учётом истории переписки.

### Ключевые возможности

| Компонент | Описание |
|---|---|
| **Векторный поиск** | `BAAI/bge-m3` через Qdrant — находит релевантные чанки по смыслу |
| **История диалога** | Каждый запрос учитывает предыдущие сообщения |
| **Кеш поиска** | LRU-кеш повторяющихся вопросов — повторный запрос возвращается мгновенно |
| **Redis-кеш** | Кеширование готовых ответов (опционально, работает без Redis) |
| **Anti-hallucination** | Если факта нет в базе — система строго отвечает «Информация отсутствует в базе знаний» |
| **Быстрый старт** | Модель прогревается при запуске сервера — первый запрос не ждёт загрузки |
| **Аналитика** | Логирование взаимодействий в `analytics.log` |

---

## Технологический стек

- **Backend**: Python 3.12, FastAPI, Uvicorn, Pydantic
- **Vector DB**: Qdrant (серверный или локальный disk-режим)
- **Embeddings**: `BAAI/bge-m3` via SentenceTransformers
- **LLM**: `g4f` — бесплатные провайдеры, без GPU и API-ключей
- **Cache**: LRU (in-memory) + Redis (опционально)
- **Frontend**: React 18, Axios

## Запуск в Replit

Проект настроен для работы напрямую в Replit без Docker:

| Воркфлоу | Команда | Порт |
|---|---|---|
| **Start Backend** | `cd backend && python -m uvicorn main:app --host localhost --port 8000` | 8000 |
| **Start application** | `cd frontend/chat_ui && npm start` | 5000 |

Запросы `/chat` из фронтенда проксируются на `http://localhost:8000` через поле `proxy` в `package.json`.

> **Первый запуск** занимает 20–30 секунд — бэкенд загружает модель `BAAI/bge-m3` из кеша и прогревает PyTorch. Порт 8000 открывается сразу; «Application startup complete» появляется после готовности модели.

## Деплой через Docker

### Требования

- [Docker](https://docs.docker.com/get-docker/) + [Docker Compose](https://docs.docker.com/compose/install/)
- [Node.js / npm](https://nodejs.org/) — для фронтенда

### Шаг 1 — Запустить инфраструктуру

```bash
cd infra
docker-compose up -d --build
```

Поднимает три сервиса:

| Сервис | Порт | Описание |
|---|---|---|
| `backend` | `8000` | FastAPI RAG API |
| `qdrant` | `6333` | Векторная база данных |
| `redis` | `6379` | Кеш готовых ответов |

Первая сборка скачивает модель `BAAI/bge-m3` (~2.2 ГБ).

### Шаг 2 — Наполнить базу знаний

```bash
pip install qdrant-client sentence-transformers
python data/upload_markdown.py data/my_documents.md --qdrant-host localhost
```

### Шаг 3 — Запустить фронтенд

```bash
cd frontend/chat_ui
npm install
npm start
```

### Переменные окружения

| Переменная | По умолчанию | Описание |
|---|---|---|
| `QDRANT_HOST` | не задан → disk-режим | Хост Qdrant-сервера |
| `REDIS_HOST` | `localhost` | Хост Redis |
| `LLM_MODEL` | `""` → `g4f.models.default` | Модель для g4f |

## Наполнение базы знаний

Поддерживаются два формата файлов: **JSON** и **Markdown**. Формат определяется автоматически по расширению.

### Формат JSON (рекомендуется)

Массив объектов. Каждый объект — один чанк документа:

```json
[
  {
    "title": "Мобильный Интернет",
    "source": "Мобильный Интернет.pdf",
    "section": "Общая информация",
    "chunk_id": 1,
    "text": "Обязательная услуга для доступа в Интернет по технологии LTE (4G), GPRS/EDGE (2G)...",
    "page": 1,
    "tags": ["internet", "activation"],
    "language": "ru"
  },
  {
    "title": "Мобильный Интернет",
    "source": "Мобильный Интернет.pdf",
    "section": "Стоимость",
    "chunk_id": 2,
    "text": "Стоимость округляется до 1 коп, объём — до 128 Кб...",
    "page": 2,
    "tags": ["internet"],
    "language": "ru"
  }
]
```

| Поле | Обязательное | Описание |
|---|---|---|
| `title` | да | Название документа (без `.pdf`) |
| `source` | нет | Имя файла-источника (по умолчанию `{title}.pdf`) |
| `section` | да | Раздел внутри документа |
| `chunk_id` | нет | Номер чанка (генерируется автоматически при загрузке) |
| `text` | да | Текст чанка — именно он векторизуется и ищется |
| `page` | да | Номер страницы в исходном PDF |
| `tags` | нет | Массив строк-тегов (или пустой массив `[]`) |
| `language` | нет | Код языка, не используется системой |

### Формат Markdown

Альтернативный формат — чанки разделяются строкой `---`:

```markdown
### [Название файла.pdf] Раздел: Название раздела (Стр. N)
**Теги:** тег1, тег2

Текст чанка — произвольный текст из документа, который будет
векторизован и загружен в базу знаний...

---
```

- **Заголовок** — название PDF-файла, раздел и номер страницы
- **Теги** — через запятую, или `нет`
- **Разделитель** — строка `---` между чанками

### Способы загрузки

#### Способ 1 — Через API (бэкенд не нужно останавливать)

```bash
# Загрузить JSON
curl -X POST http://localhost:8000/admin/upload -F "file=@my_data.json"

# Загрузить Markdown
curl -X POST http://localhost:8000/admin/upload -F "file=@my_data.md"

# Сначала очистить базу, потом загрузить новое
curl -X POST http://localhost:8000/admin/clear
curl -X POST http://localhost:8000/admin/upload -F "file=@my_data.json"
```

#### Способ 2 — Через скрипт (требует остановки бэкенда)

Бэкенд и скрипт не могут одновременно держать lock на локальное хранилище Qdrant.

**Локальный режим** (данные в `data/qdrant_storage/`):
```bash
python data/upload_markdown.py data/my_file.json
python data/upload_markdown.py data/my_file.md
```

**Docker / серверный режим** (Qdrant-сервер работает отдельно):
```bash
python data/upload_markdown.py data/my_file.json --qdrant-host localhost
python data/upload_markdown.py data/my_file.md --qdrant-host localhost --qdrant-port 6333
```

Скрипт создаёт коллекцию `knowledge_base` при первом запуске и дополняет её при повторных — старые чанки не удаляются.

## Как работает RAG-пайплайн

При каждом запросе пользователя система выполняет 4 шага:

```
Вопрос + история диалога
         │
         ▼
1. Векторный поиск      — BAAI/bge-m3 кодирует вопрос → top-3 чанка из Qdrant
   (LRU-кеш)              повторный вопрос → мгновенный ответ из кеша
         │
         ▼
2. Сборка контекста     — текст чанков, лимит 4500 символов
         │
         ▼
3. Вызов LLM (g4f)      — один запрос с системным промптом + контекстом + историей
   max 4 предложения       если ответ не в базе → «Информация отсутствует в базе знаний»
         │
         ▼
4. Очистка ответа       — убрать фразы-паразиты, обрезать до 4 предложений
         │
         ▼
Ответ + ссылки на источники (документ, раздел, страница)
```

Redis-кеш проверяется перед шагом 1 — если точный вопрос уже встречался, ответ возвращается без обращения к модели.

## API

### `POST /chat`

**Запрос:**
```json
{
  "question": "как подключить мобильный интернет?",
  "history": [
    {"role": "user", "content": "привет"},
    {"role": "assistant", "content": "Здравствуйте! Чем могу помочь?"}
  ]
}
```

**Ответ:**
```json
{
  "answer": "Для подключения мобильного интернета необходима USIM-карта...",
  "sources": [
    {"title": "Мобильный Интернет", "section": "Общая информация", "page": 1}
  ]
}
```

Поле `history` опционально — при пустом массиве запрос обрабатывается как первое сообщение.

## Известные ограничения

- **g4f** использует бесплатные провайдеры — время ответа LLM зависит от их доступности (обычно 10–40 сек)
- **Только CPU** — модель `BAAI/bge-m3` запускается без GPU; повторные запросы быстрые благодаря кешу
- **Локальный Qdrant** — одновременно только один процесс может держать lock на хранилище `data/qdrant_storage/`
