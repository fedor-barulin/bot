# Требования к базе знаний — Мотив AI

Этот документ описывает, в каком виде должны быть подготовлены материалы, чтобы бот отвечал максимально точно и не говорил «Информация отсутствует».

---

## Как работает поиск — почему важна чистота текста

При каждом вопросе пользователя система:

1. Кодирует вопрос в вектор с помощью модели `BAAI/bge-m3`
2. Находит **3 наиболее похожих чанка** из базы знаний (по косинусному сходству векторов)
3. Передаёт их текст в LLM как контекст для ответа (лимит ~4500 символов на все 3 чанка)

Модель **сравнивает смысл** вопроса со смыслом текста чанка. Это значит:

- Мусорный текст в чанке (навигация, временны́е метки, иконки) понижает релевантность при поиске
- Если нужный чанк не попал в топ-3 — бот не найдёт ответ, даже если информация в базе есть
- Если чанк слишком большой — он занимает весь контекст и вытесняет другие нужные чанки

---

## Проблемы в текущей базе знаний

Текущие чанки извлечены из корпоративного портала через OCR/скрейпинг и содержат типичные артефакты:

```
, 11:44 СврдО.Свой тариф  Разделы  Тарифные планы  Индивидуальные ...
СврдО СврдО. Свой т  Печать  Приложения №2 ...
```

Проблемы конкретно:

| Артефакт | Пример | Что делает |
|---|---|---|
| Временны́е метки | `, 11:44 СврдО.Свой тариф` | Мусор в начале каждого чанка |
| Иконки шрифта | `\ue925`, `\ue963` | Непечатаемые символы, шум для модели |
| Навигационные хлебные крошки | `Разделы › Тарифные планы › СврдО` | Не несут смысловой нагрузки |
| Дублирование заголовка | `Мобильный Интернет Мобильный Интернет` | Повторы занимают место |
| Ссылки на разделы | `см. Смена тарифа - ЛК, моб.приложение` | Мертвые ссылки, без контекста бесполезны |
| Слишком крупные чанки | до 4428 символов | Один чанк заполняет весь контекст |

## Требования к чанку

### Структура JSON-объекта

```json
{
  "title": "Название документа без расширения",
  "source": "Название документа.pdf",
  "section": "Название раздела",
  "page": 1,
  "chunk_id": 1,
  "text": "Чистый текст чанка без мусора.",
  "tags": ["тег1", "тег2"],
  "language": "ru"
}
```

### Правила для поля `text` (самое важное)

**✅ Должно быть:**

- Чистый смысловой текст — только то, что полезно сотруднику
- Законченная мысль: определение услуги, инструкция, тарифная таблица, список условий
- Язык — русский, как в оригинальном документе
- Длина: **300–900 символов** (оптимально ~500)

**❌ Нужно убрать перед загрузкой:**

- Временны́е метки и дату скрейпинга: `, 11:44`, `0.03.2026`
- Навигационные элементы: `Разделы`, `Тарифные планы`, хлебные крошки `›`
- Иконки и спецсимволы: `\ue925`, `\ue963`, `•` из шрифтов иконок
- Дублирующиеся заголовки в начале текста
- Кнопки интерфейса: `Печать`, `Приложения №2`
- Мертвые ссылки без контекста: `см. ЛК, моб.приложение`
- Номера страниц в формате `2/7 | |`

## Оптимальный размер чанка

| Параметр | Значение |
|---|---|
| Минимум | 200 символов |
| Оптимум | 400–700 символов |
| Максимум | 1000 символов |
| Жёсткий лимит системы | 4500 символов суммарно на 3 чанка |

Текущие чанки (`Свой тариф_СО` страницы 1, 4, 5, 6) по 3000–4400 символов — **это слишком много**. Один такой чанк вытесняет остальные и LLM не получает нужный контекст.

**Правило:** если страница PDF содержит несколько логически разных блоков — разбейте её на несколько чанков.

Например, страницу с тарифными условиями стоит разбить на:
- чанк «Основные параметры тарифа» (минуты, ГБ, абонплата)
- чанк «Дополнительные услуги» (безлимиты, роутер и т.д.)
- чанк «Ограничения и исключения» (Крым, роуминг)

---

## Оптимальный раздел (`section`)

Поле `section` влияет на поиск — оно добавляется к тексту при векторизации. Используйте осмысленные названия:

**✅ Хорошо:**
- `Общая информация`
- `Дополнительные услуги`
- `Стоимость и тарификация`
- `Инструкция по настройке`
- `Ограничения тарифа`
- `Контакты и поддержка`

**❌ Плохо:**
- `Стр. 3`
- `Раздел 2`
- `Прочее`

```
Дополнительные услуги в тарифе «Свой тариф»:

- Безлимитный интернет
- Режим роутера
- Безлимит на социальные сети

  (при своевременном списании абонентской платы)
- «Везде как дома» — расход пакетов интернета в национальном роуминге
- SMS на номера РФ: пакеты 50/100/200 SMS за 20/30/40 руб/мес
```

Разница: убран мусор из начала и конца, добавлен вводный заголовок (улучшает семантическое совпадение), текст структурирован.

## Покрытие базы знаний

Бот отвечает **только на основе загруженных документов**. Если тема не покрыта — бот скажет «Информация отсутствует».

Рекомендуется загрузить:

| Категория | Примеры документов |
|---|---|
| **Тарифы** | Описание каждого тарифного плана, условия, стоимость |
| **Услуги** | Мобильный интернет, SMS, роуминг, дополнительные услуги |
| **Настройка** | APN-параметры, настройка оборудования, коды USSD |
| **Поддержка** | Контакты КЦ, часы работы, сервисные номера |
| **Процедуры** | Смена тарифа, перенос номера, блокировка/разблокировка |
| **Скрипты** | Типовые ответы на частые вопросы клиентов |

## Рекомендации по подготовке

1. **Источник** — берите тексты напрямую из документов (PDF → копия текста), а не через скрейпинг веб-страниц портала
2. **Разбивка** — одна логическая тема = один чанк (не одна страница = один чанк)
3. **Заголовок чанка** — начните текст с короткого информативного предложения, которое описывает содержимое. Это помогает семантическому поиску
4. **Проверка** — после загрузки проверьте каждую тему тестовым вопросом через `/chat`
5. **Обновление** — при изменении тарифа обновляйте чанк через `/admin/clear` + `/admin/upload`

---

## Загрузка в систему

```bash
# Очистить текущую базу
curl -X POST http://localhost:8000/admin/clear

# Загрузить новый файл
curl -X POST http://localhost:8000/admin/upload -F "file=@data/clean_knowledge_base.json"
```

Загрузка занимает несколько секунд — документы векторизуются и сохраняются в Qdrant.
